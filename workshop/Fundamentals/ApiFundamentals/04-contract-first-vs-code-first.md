# Contract-First vs Code-First API Design

Two approaches to defining the shape of your API. Both are valid; each has trade-offs.

---

## Contract-First

Write the OpenAPI specification **before** writing any C# code. Generate types and interfaces from the spec.

### Workflow

```
1. Write openapi.yaml (or openapi.json)
2. Generate C# types and interfaces: dotnet tool run nswag ...
3. Implement the generated interface
4. CI validates the implementation against the spec
```

### Example: NSwag / Kiota

```yaml
# openapi.yaml
paths:
  /v1/accounts:
    post:
      operationId: createAccount
      requestBody:
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreateAccountRequest'
      responses:
        '201':
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Account'
```

```csharp
// Generated by NSwag — DO NOT EDIT
public interface IAccountsClient
{
    Task<Account> CreateAccountAsync(CreateAccountRequest body, CancellationToken ct);
}

// You implement the generated interface — compiler enforces spec conformance
public class AccountsController : ControllerBase, IAccountsClient
{
    public async Task<Account> CreateAccountAsync(CreateAccountRequest body, CancellationToken ct)
    {
        // ... implementation
    }
}
```

### Pros
- All teams (frontend, mobile, QA, backend) can work in parallel from day 1
- The spec IS the source of truth — no drift possible if enforced by CI
- Client SDKs can be generated before the server is written (Kiota / NSwag)
- Great for public APIs, external partners, and compliance-driven organisations

### Cons
- Upfront investment in writing the spec
- Spec can drift if not enforced (use Schemathesis or Dredd in CI)
- Slower initial iteration for exploratory work

## Code-First

Write the controllers and handlers first. The spec is auto-generated from the code via the built-in OpenAPI stack (.NET 9+) or Swashbuckle (.NET 8 and earlier).

### Workflow

```
1. Write the controller/handler with [ProducesResponseType] and XML <summary> comments
2. Built-in OpenAPI generates the spec from the running application
3. Scalar (or any OpenAPI UI) renders it for humans; publish the JSON to a developer portal
```

### Example: Built-in OpenAPI + Scalar (.NET 9+)

```csharp
[ApiController]
[Route("v1/[controller]")]
[ProducesResponseType<Account>(StatusCodes.Status201Created)]
[ProducesResponseType(StatusCodes.Status400BadRequest)]
public class AccountsController(IBankService svc) : ControllerBase
{
    /// <summary>Create a new bank account</summary>
    [HttpPost]
    public async Task<IActionResult> Create(
        [FromBody] CreateAccountRequest req,
        CancellationToken ct)
    {
        var account = await svc.CreateAccountAsync(req.Owner, req.InitialBalance, ct);
        return CreatedAtAction(nameof(Get), new { id = account.Id }, account);
    }
}

// Program.cs
builder.Services.AddOpenApi();       // built-in spec generation
app.MapOpenApi();                    // → GET /openapi/v1.json
app.MapScalarApiReference();         // → GET /scalar/v1  (visual UI)
```

> **On .NET 8 or earlier?** The built-in OpenAPI stack wasn't available. Swashbuckle was the de-facto standard — and it still works in .NET 9/10, it's just no longer the preferred path:
>
> ```csharp
> // Program.cs — Swashbuckle approach (.NET 8 and earlier, or when you need its ecosystem)
> builder.Services.AddSwaggerGen(c =>
> {
>     c.SwaggerDoc("v1", new OpenApiInfo { Title = "Bank API", Version = "v1" });
>     c.IncludeXmlComments(Path.Combine(AppContext.BaseDirectory, "Bank.Api.xml"));
> });
> app.UseSwagger();
> app.UseSwaggerUI();  // → GET /swagger/index.html
> ```

### Pros
- Zero spec drift — spec is always in sync with the code
- Faster iteration — especially for exploratory or single-team work
- Less upfront ceremony

### Cons
- API shape emerges during coding — harder to align multiple teams upfront
- Spec quality depends on how well developers annotate the code
- External consumers can't generate clients until the server is deployed

## Parallel Development Advantage (Contract-First)

```
Week 1: Write openapi.yaml

Week 2 onwards:
Backend  → implements IAccountsClient (generated interface)
Frontend → generates TypeScript client from spec (Kiota)
Mobile   → generates Swift client from spec
QA       → writes contract tests against the spec (Schemathesis)
Docs     → publishes spec to developer portal (Redoc / Swagger UI)

All four teams work simultaneously without blocking each other.
```

---

## Spec Drift

| Approach | Risk | Mitigation |
| :--- | :--- | :--- |
| Contract-First | Spec and code drift apart over time | CI: `schemathesis run openapi.yaml --url http://localhost:5000` |
| Code-First | Spec is always up to date | No mitigation needed |

---

## Decision Guide

```
Multiple teams / external consumers / public API / compliance requirement?
└── YES → Contract-First (NSwag / Kiota)

Single team / internal API / rapid iteration / startup?
└── YES → Code-First (built-in OpenAPI + Scalar on .NET 9+; Swashbuckle on .NET 8 and earlier) with good XML comment discipline
```

---

## In This Course

The Bank API uses **code-first** because:
- Single team, rapid iteration
- Internal API consumed by Bank.Cli (not external partners)
- `app.MapOpenApi()` generates and serves the spec (`GET /openapi/v1.json`) with zero configuration
- `Scalar.AspNetCore` adds a visual UI at `GET /scalar/v1` — no Swashbuckle required
